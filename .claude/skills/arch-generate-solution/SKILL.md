# Генерация архитектурного решения

## Задача

По исходным документам (БТ, описания, схемы процессов и др.) из папки `solutions/NNN_Название/input/`:
1. Проанализировать требования и бизнес-процесс
2. Внести изменения в Structurizr DSL (C4-архитектура)
3. Валидировать DSL
4. Сгенерировать черновик АР документа в `solutions/NNN_Название/output/`

## Аргумент

Путь к папке решения, например: `solutions/005_Формирование BPMN 2.0 и BRD`

Если аргумент не указан — перебери папки `solutions/NNN_Название/` от наибольшего номера (NNN) к наименьшему. Первая папка, у которой есть файлы в `input/`, но папка `output/` пуста или отсутствует — целевая. Если такой папки нет — спроси пользователя.

## Алгоритм

### Фаза 1. Подготовка

1. Прочитай все файлы из `<папка решения>/input/`. Если файлов несколько — прочитай все, они дополняют друг друга.
   > **BPMN-файлы (.bpmn):** Для парсинга BPMN XML используй готовый скрипт:
   > ```bash
   > python3 scripts/parse-bpmn.py <путь к .bpmn файлу>
   > ```
   > Скрипт выводит участников (participants), потоки сообщений (message flows), лейны и задачи — всё необходимое для анализа бизнес-процесса. Читать BPMN XML вручную или через Read не нужно.
   > **Большие файлы:** БТ-документы со встроенным BPMN XML часто превышают лимит токенов инструмента Read (25000). Читай такие файлы сразу через `bash` (`cat file | head -N` + `cat file | tail -n +N`), а не через Read — это избежит ошибок и повторных попыток. Если BPMN XML встроен в документ — извлеки его во временный файл и передай в `scripts/parse-bpmn.py`.
   > **BPMN-схемы и дубли в PNG/JPG:** В БТ-документах рядом с BPMN XML могут быть ссылки на дублирующие схемы в формате PNG или JPG. Эти изображения — визуальные копии тех же BPMN-схем. Читать их не нужно — достаточно разобрать BPMN в XML-формате через скрипт.
2. Прочитай текущую архитектуру:
   - `structurizr/common.dsl` — общие элементы (акторы, внешние системы)
   - `structurizr/workspace.dsl` — общая структура и includes
   - Все файлы `structurizr/domains/*/*.dsl` — модели и представления доменов
3. Определи какие системы, контейнеры и связи из входных документов уже есть в архитектуре, а какие нужно добавить.

### Фаза 2. Уточнение

**ОБЯЗАТЕЛЬНО** задай пользователю уточняющие вопросы через AskUserQuestion. Это БЛОКИРУЮЩЕЕ требование — нельзя переходить к Фазе 3 без явных ответов пользователя в ТЕКУЩЕЙ сессии.

Даже если ты уже знаешь ответы из auto-memory, MEMORY.md, предыдущих сессий или контекста проекта — **всё равно спроси**. Память может быть устаревшей, а пользователь мог изменить решение.

Вопросы:

- Альтернативные варианты, если есть
- Любые неясности из входных документов

**НЕ ПРИСТУПАЙ к изменениям без явных ответов пользователя на уточняющие вопросы в этой сессии.**

### Фаза 3. Изменения в Structurizr DSL

Внеси изменения в DSL-файлы, соблюдая соглашения из CLAUDE.md:

#### Соглашения DSL (ключевые)

- **Область видимости `!element`:** из `!element systemA` нельзя ссылаться на контейнеры другой системы — ни через `system.container`, ни по простому имени. Контейнер виден ТОЛЬКО внутри `!element` своей системы. Для кросс-системных связей одна сторона должна быть softwareSystem.
- **Кросс-системные связи контейнеров (hierarchical identifiers):** если контейнеры разных систем взаимодействуют друг с другом, помимо связей внутри `!element` блоков (container↔softwareSystem) добавь **кросс-системные связи container↔container вне `!element` блоков** через полные иерархические имена. Это обеспечивает детализацию на solution views — видно, с какими именно контейнерами другой системы идёт взаимодействие. Блок размещается в конце файла домена, после всех `!element` блоков:
  ```dsl
  // Кросс-системные связи контейнеров (для solution views)
  system_a.container_x -> system_b.container_y "Описание" "Протокол" "Теги"

  // Кросс-системные инфопотоки контейнеров
  system_b.container_y -> system_a.container_x "INFxx. Описание" "" "Dataflow,New"
  ```
  Это работает благодаря `!identifiers hierarchical` в workspace.dsl.

  > **ВАЖНО — дедупликация связей в Structurizr:** Structurizr дедуплицирует связи с **одинаковым описанием** между одними и теми же элементами (даже если одна связь system-level, а другая container-level). Из-за этого кросс-системные связи container↔container могут не отобразиться на solution view, если их описание совпадает с system-level связями из `!element` блоков.
  >
  > **Правило:** описания кросс-системных связей (и обычных, и Dataflow) должны быть **уникальными** — отличаться от описаний system-level связей в `!element` блоках. Самый простой способ — сделать описание более конкретным (упомянуть конкретный контейнер вместо системы). Пример (синхронный вызов — одна стрелка на containers, один INF на dataflow):
  > ```dsl
  > // Внутри !element system_a (system-level):
  > container_x -> system_b "Отправляет данные" "REST/HTTPS" "New"
  > system_b -> container_x "INF02. Данные из System B" "" "Dataflow,New"
  >
  > // Кросс-системные (container-level) — описания ОТЛИЧАЮТСЯ:
  > system_a.container_x -> system_b.container_y "Отправляет данные в Container Y" "REST/HTTPS" "New"
  > system_b.container_y -> system_a.container_x "INF02. Данные из Container Y" "" "Dataflow,New"
  > ```
- **autoLayout:** ландшафт и системные диаграммы — `autoLayout` (сверху вниз). Контейнерные диаграммы (containers) — `autoLayout lr` (слева направо). **Диаграммы инфопотоков (dataflow) — в противоположном направлении:** если containers `autoLayout lr`, то dataflow `autoLayout rl`; если containers `autoLayout tb`, то dataflow `autoLayout bt`. Это обеспечивает одинаковое расположение элементов на обеих диаграммах, при этом стрелки направлены противоположно (зависимости vs направление данных).
- Связи с указанием протокола: `"REST/HTTPS"`, `"kafka"`, `"TCP"`, `"MCP/HTTPS"`
- Теги: `"External"`, `"DB"`, `"Pipe"`, `"NotInProd"`, `"Dataflow"`, `"New"`, `"Changed"`
- **Теги `"New"` и `"Changed"`:** все **новые** элементы и связи, добавляемые в рамках данного решения, помечай тегом `"New"`. Все **изменяемые** (уже существовавшие, но модифицируемые) элементы и связи — тегом `"Changed"`. Эти теги визуально выделяют изменения на диаграммах (зелёная рамка/линия для New, синяя для Changed). Примеры:
  ```dsl
  genai_chatbot_service = container "GenAI чат-боты" "GenAI чат-боты" "Python" "New"
  genai_integrations = container "GenAI интеграции" "GenAI интеграции" "Java" "Changed"
  jaicp -> genai_chatbot_service "Передаёт вопросы пользователей" "REST/HTTPS" "New"
  ```
- Инфопотоки помечаются тегом `"Dataflow"` и имеют формат `"INFxx. Описание"`. Нумерация INFxx **сквозная в рамках своей системы** — чтобы на per-system dataflow view нумерация была последовательной. Например: в `!element system_a` потоки нумеруются INF01–INF05, в `!element system_b` — тоже INF01–INF03 (независимо).
- Инфопотоки между контейнерами разных систем: в каждом `!element` блоке описывай потоки на уровне system<->container (не container<->container кросс-системно), чтобы контейнеры чужих систем не «протекали» на диаграммы инфопотоков.
- **Синхронные vs асинхронные взаимодействия:**
  - **Синхронный вызов** (REST/HTTPS, MCP/HTTPS, TCP и т.п.) = **одна** стрелка на containers (от вызывающей стороны к вызываемой) и **один** инфопоток на dataflow (показывает данные, возвращаемые в ответе — от вызываемой стороны к вызывающей). Не дублируй обратной стрелкой — это загромождает диаграмму и создаёт ложное впечатление асинхронного взаимодействия.
  - **Асинхронное взаимодействие** (kafka, RabbitMQ — когда система A отправляет сообщение в B, а потом B отдельно вызывает A обратно) = **две** стрелки на containers (A→B и B→A) и **два** инфопотока на dataflow.
- **Консистентность связей и инфопотоков:** количество стрелок на диаграмме контейнеров и на диаграмме инфопотоков **должно совпадать**. Каждой связи (не Dataflow) между парой элементов должен соответствовать **ровно один** инфопоток (Dataflow). Не создавай несколько инфопотоков для пары, у которой только одна связь. Это правило применяется на ВСЕХ уровнях: внутри `!element` блоков (system↔container) и в кросс-системных связях (container↔container). **Проверяй это после создания всех связей, до валидации DSL.**
- **НИКОГДА не удалять `workspace.json`**

#### Представления (views.dsl)

Для каждой новой системы с контейнерами добавь 3 представления:
```dsl
systemContext <system> "<key>_system" "<Название> IT-система" {
    include *
    exclude relationship.tag==Dataflow
    autoLayout
}

container <system> "<key>_containers" "<Название>: системная архитектура" {
    include *
    exclude relationship.tag==Dataflow
    exclude "element.type==SoftwareSystem -> element.type==SoftwareSystem"
    autoLayout lr
}

container <system> "<key>_dataflow" "<Название>: инфопотоки" {
    include *
    exclude relationship.tag!=Dataflow
    exclude "element.type==SoftwareSystem -> element.type==SoftwareSystem"
    autoLayout rl
}
```

#### Сводный view для решения (solution view)

Если решение затрагивает контейнеры в **нескольких системах**, создай единый сводный view. Он используется в АР для единственной диаграммы контейнеров и единственной диаграммы инфопотоков.

Формат ключа: `solution_NNN_containers` и `solution_NNN_dataflow`.

Container view создаётся от любой из затронутых систем. Контейнеры всех систем включаются через `include "element.parent==<system>"`, а связанные внешние системы — через `include "->element.parent==<system>->"` ([документация](https://docs.structurizr.com/dsl/cookbook/container-view-multiple-software-systems/)):

```dsl
container <primary_system> "solution_NNN_containers" "Решение NNN: системная архитектура" {
    include "element.parent==<primary_system>"
    include "element.parent==<other_system>"
    include "->element.parent==<primary_system>->"
    include "->element.parent==<other_system>->"
    exclude relationship.tag==Dataflow
    exclude "element.type==SoftwareSystem -> element.type==SoftwareSystem"
    autoLayout lr
}

container <primary_system> "solution_NNN_dataflow" "Решение NNN: инфопотоки" {
    include "element.parent==<primary_system>"
    include "element.parent==<other_system>"
    include "->element.parent==<primary_system>->"
    include "->element.parent==<other_system>->"
    exclude relationship.tag!=Dataflow
    exclude "element.type==SoftwareSystem -> element.type==SoftwareSystem"
    autoLayout rl
}
```

### Фаза 4.5. Экспорт диаграмм в SVG

Сначала определи, затрагивает ли решение **одну** или **несколько** систем. В зависимости от этого выполняется **только один** из двух вариантов экспорта.

**Перед экспортом** закомментируй содержимое стиля `NotInProd` в `structurizr/workspace.dsl`, чтобы элементы, не выведенные в прод, не выделялись на диаграммах в документе АР:

```dsl
// Было:
element "NotInProd" {
    border dashed
    stroke "#666666"
    strokeWidth 4
    opacity 60
}

// Стало (содержимое закомментировано):
element "NotInProd" {
    // border dashed
    // stroke "#666666"
    // strokeWidth 4
    // opacity 60
}
```

#### Вариант A. Решение затрагивает одну систему — экспорт per-system views

Экспортируй диаграммы:
```bash
scripts/export-svg.sh
```

Скопируй per-system SVG-файлы из `structurizr/export/` в `<папка решения>/output/`:
- `<key>_dataflow.svg`, `<key>_dataflow-key.svg`
- `<key>_containers.svg`, `<key>_containers-key.svg`

Удали временную папку экспорта:
```bash
rm -rf structurizr/export
```

#### Вариант B. Решение затрагивает несколько систем — экспорт solution views

1. **Перенумеруй INF-потоки** в DSL-файлах так, чтобы нумерация была **сквозной по всем `!element` блокам**, затронутым в решении. Порядок: по системам в том же порядке, в котором они появляются в solution view. Например: `!element system_a` — INF01–INF05, `!element system_b` — INF06–INF08 (продолжая нумерацию).

2. **Экспортируй диаграммы**:
   ```bash
   scripts/export-svg.sh
   ```

3. **Скопируй solution SVG-файлы** из `structurizr/export/` в `<папка решения>/output/`:
   - `solution_NNN_dataflow.svg` и `solution_NNN_dataflow-key.svg`
   - `solution_NNN_containers.svg` и `solution_NNN_containers-key.svg`

4. **Верни нумерацию INF-потоков обратно** — восстанови per-system нумерацию (INF01–INFxx внутри каждой системы), чтобы DSL остался в консистентном состоянии для per-system views.

5. Удали временную папку экспорта:
   ```bash
   rm -rf structurizr/export
   ```

**После экспорта** раскомментируй содержимое стиля `NotInProd` обратно в исходное состояние (убери `// ` перед каждой строкой внутри блока).

#### Верификация консистентности диаграмм containers и dataflow

После копирования SVG-файлов в output **обязательно** проверь консистентность с помощью готового скрипта:

```bash
bash scripts/verify-svg-consistency.sh <папка решения>/output/*_containers.svg <папка решения>/output/*_dataflow.svg
```

Скрипт подсчитывает стрелки (по протоколам `[REST/HTTPS]`, `[MCP/HTTPS]` и т.д.) на containers и уникальные INF-коды на dataflow. Если количества не совпадают — скрипт завершится с кодом 1.

**Если количества не совпадают** — типичные причины и исправления:
1. **Лишние инфопотоки (INF больше, чем стрелок на containers):** для пары элементов создано несколько инфопотоков (запрос + ответ), но только одна связь. Для синхронных вызовов должна быть **одна** связь и **один** инфопоток — убери лишний.
2. **Лишние обратные стрелки на containers:** для синхронного вызова (REST, MCP) создана обратная связь B→A помимо прямой A→B. Убери обратную — синхронный вызов отображается одной стрелкой от вызывающей стороны.
3. **Кросс-системные Dataflow-связи не отображаются на solution view:** Structurizr дедуплицирует связи с одинаковым описанием. Если кросс-системная связь container↔container имеет то же описание, что и system-level связь из `!element` блока — измени описание кросс-системной связи, сделав его уникальным (например, упомяни конкретный контейнер вместо системы).
4. **System-level связь отображается вместо container-level:** на solution view обе системы раскрыты до контейнеров, system-level связи могут быть подавлены. Убедись, что для каждой system-level связи есть кросс-системный container-level аналог с **уникальным** описанием.

После исправления — переэкспортируй SVG и проверь повторно.

#### Обработка легенд (key SVG)

После копирования отредактируй SVG-файлы легенд (`*-key.svg`), чтобы уменьшить элементы:
- Уменьши все размеры в **2 раза**: rect, font-size, tspan dy, rx/ry, stroke-width — всё пропорционально. Точные исходные размеры могут отличаться в разных файлах — главное сохранить коэффициент ×0.5 (например, rect `450x300` → `225x150`, font-size `30px` → `15px`, tspan dy `40px` → `20px`)
- В одной строке размещай **до 8 элементов** с шагом ~250px по горизонтали. Если элементов больше 8 — переноси остальные на следующую строку
- Стрелки (Dataflow, Relationship и др.) тоже уменьши пропорционально и сдвинь по вертикали для визуального выравнивания с прямоугольниками
- Обнови viewBox под фактическое содержимое: ширина = min(кол-во элементов, 8) × 250 + отступы, высота ≈ 200 на каждую строку

#### Подготовка: примеры из существующих решений
